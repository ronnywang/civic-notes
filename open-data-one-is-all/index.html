根本原因

這不是態度問題，是架構問題

要求各機關「上傳得更好」解決不了問題，因為問題的根源是「各機關自行上傳」這個設計本身。

現況——分散式上傳

每個機關各自決定格式、欄位、更新頻率

  • 沒有人有責任確認別人上傳的資料是否可用
  • 格式選擇由機關自填，平台不驗證實際內容
  • 同一套格式在不同機關標為 XML 或 ZIP，使用者無從篩選
  • 沒有人追蹤「連結是否失效」、「去年有傳今年沒傳」
  • 縣市政府缺席，沒有機制可以施壓
破局點——可見度

把「協調難題」轉換成「可見度問題」

  • 資料既然公開發布，就可以被公開評估——不需要機關同意
  • 自動化驗證：程式每日跑一次，結果客觀可重現
  • 地圖公開後，機關的選項只有兩個：改善，或承擔紅燈的難堪
  • 地方議員有政治動機拿這張地圖去質詢縣長，無需額外說服

在 data.gov.tw 建立「一即是全」合規監測專區

最關鍵的發現

標準格式早已存在，問題只是沒有統一入口

183 個中央機關已在使用同一套主計系統標準格式——有 manifest 目錄、有 schema 欄位定義、欄位名稱統一。

183
中央機關已採用主計系統標準 ZIP 格式，含 manifest.csv 與統一欄位定義
11,136
筆資料集符合此標準格式——散落在 XML 和 ZIP 兩種標示下，使用者難以發現
0
採用此標準的縣市政府——中央與地方完全斷層，且主計總處本身反而是死連結

這代表問題不是「技術上做不到」，而是「沒有統一入口、沒有可見度、沒有壓力」。數發部不需要發明新標準——只需要建立一個讓現有標準被看見、缺席者被看見的平台。

各方角色

數發部來主導，法務部只需做一件事

這個提案的設計原則：讓每個機關只需要做它能做的那件事，不需要跨機關協調。

數位發展部 · 主責

建立「一即是全」合規監測專區

  • 在 data.gov.tw 建立各主題（預算／地價／法規）的合規地圖，交通燈顯示各機關狀態
  • 以程式自動驗證：URL 可用性、格式正確性、更新頻率、欄位完整性
  • Beta 版直接對已公開資料端點跑驗證，不需要任何機關同意
  • 先以現有可明確定義的標準做出第一版，讓「函釋待定」欄位空著——用存在本身施壓法務部
  • 開放 API 讓議員助理、公民記者、NGO 可以嵌入比較圖
核心優勢：資料既然公開發布，就可以被公開評估。數發部無需說服任何機關——地圖上線，機關自會感受到壓力。
法務部 · 一次性任務

訂定函釋與地方法規的資料標準

  • 定義函釋的資料格式與欄位標準
  • 定義地方自治法規的機器可讀規範
  • 標準訂完，責任結束——執行由數發部的合規地圖自動追蹤
現有中央法規（法規命令）的標準已可沿用，先讓地圖上線，函釋與地方法規留空，標注「等待法務部訂定標準中」。

推進路線

三個階段，第一階段數發部可以獨立啟動

第一階段
以現有標準建出初版地圖
用中央法規、法規命令的現有標準，把預算、地價、法規三個主題的合規地圖建出來。函釋和地方法規欄位明顯標注「待訂定」。數發部可獨立啟動，不需要法務部先動。
數發部獨立啟動
第二階段
地圖的存在倒逼法務部訂標準
地圖上線後，「函釋待訂定」的空白欄位是一個持續的公開壓力。地方議員可以用預算和地價的比較地圖去質詢縣長，製造地方施壓動能。
借助外部壓力
第三階段
函釋標準完成，地圖自動擴充
法務部訂完函釋和地方法規標準後，合規地圖自動加入驗證邏輯，不需要人工介入。整個系統進入自動化運作。
自動擴充
